# ACCEPTANCE CRITERIA

This document defines what qualifies as valid completed work.

No task is considered complete simply because code exists.

Completion requires:
- verification
- validation
- review
- confidence
- documentation

---

# Core Principle

A task is COMPLETE only when:
1. requirements are satisfied
2. implementation exists
3. behavior is validated
4. major risks are understood
5. documentation is updated
6. Manager confidence >=95%

Anything less is NOT complete.

---

# Minimum Completion Rules

Every meaningful task must include:

- clear goal
- actual implementation
- verification/testing
- reviewer validation when needed
- proper handoff
- confidence score
- known limitations

---

# Coding Task Criteria

A coding task is COMPLETE only if:

1. required files exist
2. code is readable
3. code matches sprint goal
4. major syntax/runtime errors checked
5. build/test status reported
6. changed files documented
7. no hidden assumptions
8. Reviewer concerns addressed
9. Manager approves

---

# Testing Task Criteria

A testing task is COMPLETE only if:

1. tests/checks were actually run
2. commands executed are listed
3. results are documented honestly
4. failures are explained clearly
5. root causes identified when possible
6. confidence score included

No fake “tests passed” claims allowed.

---

# Review Task Criteria

A review is COMPLETE only if:

1. architecture evaluated
2. maintainability evaluated
3. readability evaluated
4. scalability risks discussed
5. security concerns discussed
6. confidence score assigned
7. approval/rejection clearly stated

---

# Documentation Task Criteria

Documentation is COMPLETE only if:

1. files are updated
2. information is structured
3. summaries are understandable
4. important decisions preserved
5. obsolete info cleaned when needed

---

# Scraping/Research Task Criteria

Research is COMPLETE only if:

1. sources are listed
2. source reliability considered
3. assumptions separated from facts
4. summaries are structured
5. important findings documented

---

# Confidence Rules

Confidence must be estimated honestly.

95–100%
- safe to proceed

80–94%
- usable but concerns remain

Below 80%
- not approved
- requires clarification, testing, or rework

Confidence inflation is forbidden.

---

# Rework Rules

Rework required if:
- acceptance criteria failed
- confidence <95%
- reviewer rejects
- tester detects major issue
- architecture weak
- implementation incomplete
- task drift occurred

---

# Hidden Failure Rule

Agents must never:
- hide incomplete work
- pretend implementation is production-ready
- fake confidence
- skip verification silently

---

# Sprint Completion Rules

A sprint is COMPLETE only if:

1. all critical tasks resolved
2. testing completed or documented
3. reviewer approved
4. organizer updated memory/logs
5. manager confidence >=95%
6. blockers documented

---

# Final Rule

"Implemented" does NOT mean "Completed."

"Completed" requires:
Implementation
→ Verification
→ Review
→ Approval
→ Documentation
